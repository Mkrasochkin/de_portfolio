# Международный финтех: ETL-пайплайн для аналитики транзакций

## Описание проекта

В рамках проекта был реализован ETL-пайплайн для международной финтех-компании, предоставляющей услуги мгновенных денежных переводов между странами. Система собирает данные из PostgreSQL, переносит их в аналитическое хранилище Vertica и формирует витрину бизнес-метрик. Суть проекта: необходимость в сборе и подготовке данных о финансовых операциях для последующей аналитики. Цель — выявить динамику общего оборота компании и факторы, влияющие на неё.


## Архитектура данных

![Архитектура проекта](https://github.com/Mkrasochkin/de_portfolio/blob/main/12_project_fintech_etl_pipeline/project_1749138984.png) 


Проект реализован по классической схеме DWH с использованием современных инструментов обработки данных: 
1. Источники (Sources): PostgreSQL — транзакционные данные и курсы валют.
2. Оркестрация: Apache Airflow — управление зависимостями, расписанием и инкрементальной загрузкой.
3. Хранилище (MPP): Vertica — использование слоев STAGING и DWH для высокопроизводительной аналитики. 
4. Витрина данных(CDM): Финальная витрина `global_metrics`, рассчитанная с применением бизнес-правил.




## Навыки и инструменты

- **Python/Airflow**: Использование `PostgresHook`, `VerticaHook` и `PythonOperator`.
- **Vertica (MPP DB)**: Высокопроизводительное хранилище. Настроены проекции, сегментация и партиционирование.
- **SQL**: Написание DDL и DML запросов, включая сложную агрегацию и расчет валютных курсов.
- **Pandas**: Первичная обработка и буферизация данных при переносе между системами.




## Ключевые особенности реализации

### 1. Высокопроизводительное хранилище (Vertica)

В отличие от стандартных реляционных БД, в этом проекте Vertica настроена на максимальную скорость аналитики:

- **Сегментация**: Использован `SEGMENTED BY HASH(transaction_dt, operation_id)` для равномерного распределения нагрузки на узлы кластера.
- **Проекции**: Созданы кастомные проекции с сортировкой (`ORDER BY transaction_dt`), что ускоряет ежедневные агрегации в десятки раз.
- **Партиционирование**: Реализовано деление по датам (`PARTITION BY DATE`), что позволяет эффективно управлять данными и поддерживать инкрементальную загрузку.

### 2. ETL Пайплайн: Извлечение и загрузка (`1_data_import.py`)

- **Инкрементальность**: Скрипты используют `execution_date`, загружая данные за «вчерашний» день.
- **Идемпотентность**: Пайплайн можно безопасно перезапускать — данные не дублируются.
- **Качество**: На этапе экстракции фильтруются только успешные транзакции (`status = 'done'`).

### 3. Бизнес-логика и Витрина (`2_datamart_update.py`)

Второй этап пайплайна формирует витрину `global_metrics`.

- **Конвертация валют**: Все суммы приводятся к единому знаменателю через `JOIN` с таблицей курсов по дате и коду валюты.
- **Очистка данных**: Исключение тестовых аккаунтов (`account_number_from < 0`).



## Общий вывод

Был реализован ETL-пайплайн для финтех-стартапа, обеспечивающий сбор и подготовку данных о международных платежных операциях. Сформирована аналитическая витрина с ежедневными метриками, отражающими обороты компании и поведение пользователей. Инфраструктура организована с использованием PostgreSQL, Vertica и Airflow, что позволяет стабильно обновлять данные и обеспечивать своевременный анализ ключевых показателей.
