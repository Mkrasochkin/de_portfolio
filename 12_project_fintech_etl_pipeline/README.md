# Международный финтех: ETL-пайплайн для аналитики транзакций

## Описание проекта
Проект по разработке промышленного ETL-пайплайна для международного финтех-стартапа. Система автоматизирует сбор данных из децентрализованных PostgreSQL-источников в аналитическое хранилище Vertica (MPP) для ежедневного расчета ключевых бизнес-метрик и формирования аналитических витрин. 


Проект по созданию масштабируемого ETL-пайплайна для финтех-стартапа. Система собирает данные из децентрализованных источников (PostgreSQL), переносит их в аналитическое хранилище (Vertica) и формирует витрину бизнес-метрик.

Этот проект представляет собой разработку отказоустойчивого и масштабируемого ETL-пайплайна для международного финтех-стартапа. Система автоматизирует сбор данных из децентрализованных PostgreSQL-сервисов в аналитическое хранилище Vertica для расчета ключевых бизнес-метрик. 



В рамках учебного проекта был реализован ETL-пайплайн для международной финтех-компании, предоставляющей услуги мгновенных денежных переводов между странами. Проект охватывает сбор, обработку и анализ транзакционных данных с целью выявления тенденций и ключевых метрик.
Суть проекта:
Необходимость в сборе и подготовке данных о финансовых операциях для последующей аналитики.
Цель — выявить динамику общего оборота компании и факторы, влияющие на неё.


# Архитектура данных

![Архитектура проекта](https://github.com/Mkrasochkin/de_portfolio/blob/main/11_project_cloud_technologies/2_1747338249.png) 

Проект реализован по классической схеме DWH с четким разделением ответственности:

1. **Sources (OLTP)**: Внешние базы данных PostgreSQL, содержащие транзакции и справочники курсов валют.
2. **Staging (STG)**: Слой сырых данных в Vertica. Оптимизирован для быстрой вставки (использование проекций и сегментации по хешу).
3. **Data Mart (CDM/DWH)**: Финальная витрина `global_metrics`, рассчитанная с применением бизнес-правил.
4. **Orchestration**: Apache Airflow — управление расписанием, обработка исторических данных (Backfill) и мониторинг.

Архитектура проекта 
Проект реализован по классической схеме DWH с использованием современных инструментов обработки данных: 
1. Источники (Sources): PostgreSQL — транзакционные данные и курсы валют.
2. Оркестрация: Apache Airflow — управление зависимостями, расписанием и инкрементальной загрузкой.
3. Хранилище (MPP): Vertica — использование слоев STAGING и DWH для высокопроизводительной аналитики.

# 🚀 Ключевые особенности реализации

## 1. Высокопроизводительное хранилище (Vertica)

В отличие от стандартных реляционных БД, в этом проекте Vertica настроена на максимальную скорость аналитики:

- **Сегментация**: Использован `SEGMENTED BY HASH(transaction_dt, operation_id)` для равномерного распределения нагрузки на узлы кластера.
- **Проекции**: Созданы кастомные проекции с сортировкой (`ORDER BY transaction_dt`), что ускоряет ежедневные агрегации в десятки раз.
- **Партиционирование**: Реализовано деление по датам (`PARTITION BY DATE`), что позволяет эффективно управлять данными и поддерживать инкрементальную загрузку.

## 2. ETL Пайплайн: Извлечение и загрузка (`1_data_import.py`)

- **Инкрементальность**: Скрипты используют `execution_date`, загружая данные строго за «вчерашний» день.
- **Идемпотентность**: Пайплайн можно безопасно перезапускать — данные не дублируются.
- **Качество**: На этапе экстракции фильтруются только успешные транзакции (`status = 'done'`).

## 3. Бизнес-логика и Витрина (`2_datamart_update.py`)

Второй этап пайплайна формирует витрину `global_metrics`. Ключевые правила:

- **Конвертация валют**: Все суммы приводятся к единому знаменателю через `JOIN` с таблицей курсов по дате и коду валюты.
- **Очистка данных**: Исключение тестовых аккаунтов (`account_number_from < 0`).
- **Метрики**:
  - `amount_total` — суммарный оборот в единой валюте.
  - `cnt_transactions` — количество операций.
  - `avg_transactions_per_account` — средний чек/активность.
  - `cnt_accounts_make_transactions` — число уникальных активных клиентов.

Сформирована аналитическая витрина с ежедневными метриками, отражающими обороты компании и поведение пользователей. Инфраструктура организована с использованием PostgreSQL, Vertica и Airflow, что позволяет стабильно обновлять данные и обеспечивать своевременный анализ ключевых показателей. Проект демонстрирует работоспособность предложенного решения и готовность к промышленному применению.

### Структура репозитория
Файлы в репозитории будут использоваться для проверки и обратной связи по проекту. Поэтому постарайтесь публиковать ваше решение согласно установленной структуре: так будет проще соотнести задания с решениями.

Внутри `src` расположены папки:
- `/src/dags` - вложите в эту папку код DAG, который поставляет данные из источника в хранилище. Назовите DAG `1_data_import.py`. Также разместите здесь DAG, который обновляет витрины данных. Назовите DAG `2_datamart_update.py`.
- `/src/sql` - сюда вложите SQL-запрос формирования таблиц в `STAGING`- и `DWH`-слоях, а также скрипт подготовки данных для итоговой витрины.
- `/src/py` - если источником вы выберете Kafka, то в этой папке разместите код запуска генерации и чтения данных в топик.
